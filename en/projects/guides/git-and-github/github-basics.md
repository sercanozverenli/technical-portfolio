# GitHub Foundations and Repository Architecture

This document covers the architectural logic of the GitHub platform, configuration processes for remote server repositories, the security/cost dynamics of Public and Private repositories, authentication protocols (HTTPS vs. SSH), and GitHub interface components in minute detail.

---

## 1. What is GitHub and Why is it Necessary?

Git is a distributed version control system that tracks the version history of your project on your local computer. GitHub, on the other hand, is a web-based platform that allows for the hosting, backing up, and management of collaboration processes for these local Git repositories in the cloud (remote).

### Core Pillars:
* **Centralized Distributed Structure:** If your computer suffers hardware damage, the entire commit history of the project is fully protected in the cloud.
* **Asynchronous Work:** Multiple developers can work simultaneously on different parts of the same codebase without overwriting each other's code.
* **Open Source Ecosystem:** It is the hub of the culture of contributing to projects worldwide (Open Source Contribution) and code review (Code Review).

## 2. Visibility and Access Architecture: Public vs. Private

The visibility level chosen when creating a repository on GitHub directly affects the project's access permissions, cost policy, and automation limits.

| Feature / Metric | Public Repository | Private Repository |
| :--- | :--- | :--- |
| **Visibility** | Anyone on the internet can examine the code. | Visible only to the repository owner and invited partners (Collaborators). |
| **Write/Modify Permission** | Only authorized users with permission can push. | Only explicitly invited authorized users can push. |
| **Search Engine Index** | Indexed on Google and GitHub Explore. | Completely hidden; search engines cannot scan it. |
| **GitHub Actions Cost** | Completely free and has unlimited minutes. | Limited to 2000 minutes per month for free accounts. |
| **Security Scans** | Dependabot and Secret scans are free. | Paid plan required for advanced corporate scans. |

### Security Warning
The fact that code is readable by everyone in Public repositories can pose a major security risk. `.env` files, API keys (AWS, Firebase, etc.), or database passwords accidentally committed can be scanned and seized by automated bots in seconds. Such files must absolutely be included in `.gitignore` before being pushed to the repository.

## 3. Initial Repository Setup and Configuration Files

When creating a new repository, three core files are included in the architecture to ensure the system starts clean and in accordance with standards:

### A. README.md (Project Showcase)
Written in Markdown format, this file is displayed on the project's main page. It is the first place a developer or potential employer reads when they encounter the project. A good README.md should include:
* The purpose of the project and what it does.
* Dependencies (Requirements) and installation steps (Installation).
* Running commands (Usage).
* Contributing guidelines (Contributing).

### B. .gitignore Templates
Depending on the project type (Python, Node.js, Java, C++, etc.), it prevents temporary files generated by the local operating system or compilers (`node_modules/`, `__pycache__/`, `.DS_Store`, `.env`) from being sent to the remote server. The healthiest method is to select the relevant language on the repository creation screen to have it generated automatically.

### C. Licensing (License) Architecture
It establishes the legal ground for copyright and usage rights of the code in Public projects. All rights of a project without a selected license remain reserved, and others cannot legally copy and use the code.
* **MIT License:** The most common and flexible license. It allows the code to be used for any purpose, including commercial, provided that the author's name and original license text are preserved.
* **GNU GPLv3:** A strict open-source license. It mandates that derivative projects developed using this code must also be open-source and GPLv3 licensed.
* **Apache 2.0:** Similar to MIT, but additionally provides patent rights protection to the user and requires major changes to be noted.

## 4. Authentication Protocols: HTTPS vs. SSH

A secure tunnel must be established for the Git terminal on your local computer to transfer data (push/pull) to GitHub cloud servers. GitHub has completely banned logging in with plain-text passwords.

### 1. HTTPS and Personal Access Token (PAT)
* **Protocol Address:** `https://github.com/username/repo-name.git`
* **Operating Logic:** Instead of your GitHub account password, a timed, special-permission password string (Token) generated from the interface is used.
* **Use Case:** Suitable for situations requiring quick setup and scenarios where SSH ports (Port 22) are blocked by corporate networks.

### 2. SSH (Secure Shell) - Cryptographic Validation
* **Protocol Address:** `git@github.com:username/repo-name.git`
* **Operating Logic:** Two keys are generated on your computer with asymmetric encryption: one private (Private Key) and one open (Public Key). The Public key is defined on GitHub. Your local computer is verified in the background without a password for every transaction.

#### Step-by-Step SSH Key Creation and Configuration Commands:

**Step 1:** Generate a new SSH key pair using Ed25519, the current and most secure algorithm:
```bash
ssh-keygen -t ed25519 -C "user_email@example.com"
```
*(After the command, the system will ask for the file location and a passphrase. You can proceed with default settings by pressing Enter directly.)*

**Step 2:** Activate the SSH agent (ssh-agent) running in the background:
```bash
eval "$(ssh-agent -s)"
```

**Step 3:** Register the generated private key to this agent:
```bash
ssh-add ~/.ssh/id_ed25519
```

**Step 4:** Copy the generated public key and paste it into the `Settings -> SSH and GPG keys -> New SSH Key` field in your GitHub profile:
```bash
cat ~/.ssh/id_ed25519.pub
```

**Step 5:** Test that the connection is working properly via the terminal:
```bash
ssh -T git@github.com
```
*If successful, the following output should be received in the terminal:*
`Hi username! You've successfully authenticated, but GitHub does not provide shell access.`

## 5. Global Identity Settings Before Initial Synchronization

Before committing and pushing to the remote server for the first time, you must configure global settings from the terminal so that your name and e-mail address appear correctly in the Git history:

```bash
# Define your name as it appears on your GitHub account
git config --global user.name "Your Name"

# Define your e-mail address linked to your GitHub account
git config --global user.email "user_email@example.com"

# Verify the configured settings by listing them
git config --list
```

## 6. GitHub Interface Anatomy and Functions of Tabs

The tabs located in the top menu of a GitHub repository are the project's management panels:

1.  **Code:** The main screen containing the project's active file tree, branch list, commit history, and release versions.
2.  **Issues:** A ticket system where bugs, new features to be added, or tasks to be done in the project are listed, labeled, and assigned.
3.  **Pull Requests (PR):** A request to merge changes made from a different branch or forked repository into the main codebase. Code reviews take place entirely here.
4.  **Actions:** The automation center that manages the project's CI/CD (Continuous Integration / Continuous Deployment) processes. Setting up automatic tests when code is pushed is configured here.
5.  **Projects:** Integrated Trello-like Kanban boards that allow managing Issues and Pull Requests cards in accordance with agile (Agile/Scrum) methodologies.
6.  **Wiki:** An encyclopedic sub-page where comprehensive usage documentation, guides, and architectural details about the project are written.
7.  **Security:** The section that scans for security vulnerabilities in dependencies (Dependabot), catches secret keys leaked into the code, and manages security policies.
8.  **Settings:** The management center where administrative operations such as changing the repository name, converting its visibility (Public/Private), giving access authority to teams, defining branch protection rules, or deleting the repository completely are performed.
